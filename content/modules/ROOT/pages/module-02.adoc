= Assurance

*Sovereignty isn’t just about control — it’s about confidence*

Confidence that your applications and clusters operate exactly as intended, fully compliant with regulations, security standards, and internal policies no matter where they run.

Without robust assurance, even the most advanced cloud native environments carry hidden risks. A single compliance gap can expose your organization to regulatory fines, security breaches, data sovereignty violations, or loss of customer trust. Assurance eliminates that uncertainty. It delivers verifiable proof that every workload and cluster meets the highest standards of security and compliance across EMEA, the US, and beyond.

*Achieving this level of assurance requires a practical and structured approach.*

So let's get practical in our example of an EMEA environment and a US environment and observe, audit, police and manage our clusters and application applications. We will start by looking at the security, compliance and environment insights that we can get from Red Hat Advanced Cluster Security for Kubernetes (RHACS).

== Working with RHACM Policies to Secure your Infrastructure

Traditional approaches to multi-cluster compliance rely on manual configuration, per-cluster runbooks, or bespoke automation scripts. These approaches are error-prone, difficult to audit, and do not scale. RHACM solves this by providing:

* **Declarative policy-as-code** — compliance requirements are expressed as Kubernetes resources that can be version-controlled, reviewed, and audited.
* **Automatic placement** — policies are delivered to the correct clusters based on ClusterSet membership and labels, eliminating manual targeting.
* **Continuous compliance monitoring** — RHACM continuously evaluates policy compliance and surfaces violations in a centralized dashboard.
* **Separation of concerns** — platform teams define the governance framework, while application teams operate within the guardrails.

By the end of this module, you will be able to:

* Author self-contained policy files with embedded Placements and PlacementBindings
* Use Placements that combine the `managed` ClusterSet with `location` label selectors
* Enforce region-specific compliance controls (GDPR for EMEA, NIST for US)
* Organize policies into PolicySets for governance visibility
* Verify compliance status via the RHACM console

== Deploying Policies to Manage Clusters

We will start by deploying policies to the EMEA Cluster. The EMEA - European Union's - General Data Protection Regulation (GDPR) is one of the most comprehensive data protection frameworks in the world. It applies to any organization that processes personal data of EU residents, regardless of where the organization is headquartered. For Kubernetes clusters operating in EMEA, GDPR imposes specific technical requirements that must be codified as policy.

The three EMEA policies in this module address the core pillars of GDPR's technical requirements: **data residency labeling** (Articles 44-49 — ensuring sovereignty namespaces exist with proper classification metadata), **data protection by design** (Article 25 — network isolation), and **security of processing** (Article 32 — pod security hardening).

=== Deploying Policies to the EMEA Managed Cluster

GDPR Articles 44-49 restrict the transfer of personal data outside the European Economic Area (EEA) unless specific safeguards are in place. Before any technical control can be applied — network policies, pod security, encryption — there must be a **clearly defined, properly labeled environment** that signals its compliance status to both automated systems and human auditors.

Namespace labels and annotations serve as the foundation of Kubernetes-native compliance. They enable:

* **Automated policy targeting** — other policies (like network restrictions or pod security) can use namespace selectors to apply controls only to namespaces marked as GDPR-restricted, avoiding unnecessary restrictions on non-sensitive workloads.
* **Audit and reporting** — regulators and internal compliance teams can query the cluster to identify all namespaces handling EU personal data. Annotations provide human-readable context such as the applicable GDPR articles, the compliance framework, and the data protection officer contact.
* **Workload placement governance** — by requiring labels like `sovereignty: emea-only` and `data-classification: gdpr-restricted`, organizations establish a contract: any workload deployed to this namespace is subject to GDPR controls, and any workload handling EU personal data _must_ be deployed to a namespace with these labels.

This policy ensures two sovereignty namespaces exist on every EMEA cluster:

* **`gdpr-workloads`** — the primary GDPR-controlled namespace, fully labeled with data classification, sovereignty, compliance framework, and Pod Security Admission labels. Annotations document the applicable regulation, specific GDPR articles, data residency restrictions, and the DPO contact.
* **`data-residency-emea`** — an additional data residency namespace with core GDPR labels and annotations, providing a dedicated landing zone for workloads with strict EU-only data residency requirements.

Without these labeled namespaces, organizations cannot demonstrate to regulators that they have designated environments for EU personal data processing — a fundamental GDPR compliance gap.

Lets Apply another policy to the EMEA Cluster. This policy implements **network-level data sovereignty** by creating a NetworkPolicy in the `data-residency-emea` namespace that blocks all egress traffic to external IP addresses. Only cluster-internal communication (RFC 1918 addresses) and DNS resolution are permitted. This ensures that:

* Workloads in the GDPR-restricted namespace **cannot send data to external endpoints** — whether intentionally (API calls to non-EU services) or accidentally (misconfigured logging, telemetry, or dependency resolution).
* The only allowed communication is **within the cluster boundary**, which is presumed to be within the EU/EEA geographic region.
* DNS resolution is permitted so that internal service discovery continues to function.

=== Deploying Policies to the US Managed Cluster

Now that we have locked down the *hcp-emea* cluster, lets work on locking down the *aws-us* cluster with two Policies. 

The National Institute of Standards and Technology (NIST) Special Publication 800-53 is the primary security and privacy controls framework used by US federal agencies and many private sector organizations. Unlike GDPR, which focuses heavily on data residency and individual privacy rights, NIST 800-53 takes a broader systems-security approach — emphasizing access control, audit accountability, boundary protection, and continuous monitoring.

While GDPR asks "where is the data and can it leave?", NIST 800-53 asks "who can access the system, is access being logged, and are the boundaries controlled?" This difference in philosophy results in policies that address similar sovereignty concepts but with different technical implementations.

The two US policies in this module address: **namespace compliance labeling** (SC-16) and **audit accountability** (AU-2).

Audit logging is the cornerstone of the NIST compliance model. Where GDPR relies on technical prevention (blocking data from leaving), NIST relies on **detection and accountability** — the assumption that if all actions are logged and monitored, unauthorized activity will be identified and responded to. This is why the US network boundary policy (Step 5) allows outbound HTTPS while this audit policy ensures that all API server activity is captured.

This policy ensures two things:

* The OpenShift API Server audit profile is set to `WriteRequestBodies`, which captures the full content of all write requests (creates, updates, patches, deletes) to the API server. This provides a complete audit trail of who changed what and when — essential for incident response, forensic analysis, and regulatory compliance reporting.
* A dedicated `audit-logging` namespace exists for centralized log collection infrastructure, labeled with `compliance-framework: nist-800-53` to indicate its purpose.

=== Key Takeaways

. **Self-contained policy files** — each YAML includes the Policy, Placement, and PlacementBinding, reducing complexity and making each policy independently deployable.
. **`managed` ClusterSet + location labels** — Placements reference the `managed` ClusterSet and use `location` label selectors for precise regional targeting.
. **Symmetric policy coverage** — both regions have 3 region-specific policies + 1 shared baseline, enabling direct compliance comparison.
. **Different frameworks, different controls** — the same sovereignty concepts (data governance, networking, runtime/audit) are addressed with different technical implementations per jurisdiction.
. **Inform mode for visibility** — all policies use `remediationAction: inform`, providing compliance visibility without automatically changing cluster state. This allows teams to assess compliance posture before switching individual policies to `enforce` mode.

== Maintaining Control in Sovereign and Multi-Region Environments

In sovereign cloud setups, strict control over what runs inside your infrastructure is essential for data protection, regulatory compliance, and operational independence.

Red Hat Advanced Cluster Security (RHACS) includes built-in policies that detect and block risky behaviors such as unauthorized access attempts, persistent threats, lateral movement, and data exfiltration. These policies help you enforce the immutable container principle — containers should only run exactly what you approved, with no unexpected changes.

[[use-policy-as-code-block-anomalous-runtime]]

== Use Policy-as-Code to Block Anomalous Behavior at Runtime

Package managers such as apt, apk, yum, and dnf are useful on virtual machines, but they have no place in production containers. Allowing them enables uncontrolled software installation, dramatically increases the attack surface, and breaks auditability and immutability — all of which are core requirements for sovereign cloud environments.

Bottom line: No package manager execution means no unauthorized changes — resulting in more secure, more auditable, and more compliant containers.

Now, our security and operations teams want simplifed workflows and flexibility in where to deploy their applications. Policy-as-Code is a great way to achieve this.

*Procedure*

. Switch to the *Bastion* tab on the right.
. Verify the pods are running in the patient-portal namespace:
+
[source,sh,role=execute]
----
oc get pods -n patient-portal
----

. Get the frontend pod name and exec into the container:
+
[source,sh,role=execute]
----
POD=$(oc get pod -n patient-portal -l app=frontend -o jsonpath="{.items[0].metadata.name}")
oc exec -n patient-portal $POD -it -- /bin/sh
----

. Check the operating system release information:
+
[source,sh,role=execute]
----
cat /etc/os-release
----
+
[NOTE]
====
If you see */home/fritz $* you've confirmed you have a shell and access to the frontend application.
====

. Since this is an alpine image, we can see the package manager is apk. Try to run the package manager update command:
+
[source,sh,role=execute]
----
apk update
----
+
NOTE: OpenShift enforces strict security policies by default (via the restricted Security Context Constraint or SCC).

. Exit from the container (type `exit` or `Ctrl-D`)and return to the bastion host terminal.
. Then, create a security policy (As code) to enforce runtime protection against package manager execution. Apply the following policy configuration:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc apply -f - <<EOF
---
apiVersion: config.stackrox.io/v1alpha1
kind: SecurityPolicy
metadata:
  name: alpine-linux-package-manager-execution-patient-portal-enforcement
  namespace: stackrox
spec:
  policyName: Alpine Linux Package Manager Execution - Patient-Portal Enforcement
  description: Alert when the Alpine Linux package manager (apk) is executed at runtime
  rationale: Use of package managers at runtime indicates that new software may be being introduced into containers while they are running.
  remediation: Run 'apk --purge del apk-tools' in the image build for production containers. Change applications to no longer use package managers at runtime, if applicable.
  categories:
    - Anomalous Activity
    - Package Management
  lifecycleStages:
    - RUNTIME
  eventSource: DEPLOYMENT_EVENT
  severity: LOW_SEVERITY
  enforcementActions:
    - FAIL_KUBE_REQUEST_ENFORCEMENT
    - KILL_POD_ENFORCEMENT
  policySections:
    - policyGroups:
        - fieldName: Process Name
          booleanOperator: OR
          values:
            - value: apk
            - value: apk-tools
            - value: apk update
            - value: apk add
            - value: apk remove
            - value: apk upgrade
            - value: apk del
            - value: apk cache
            - value: apk clean
            - value: apk info
            - value: apk show
            - value: apk version
  mitreAttackVectors:
    - tactic: TA0011
      techniques:
        - T1105
  criteriaLocked: false
  mitreVectorsLocked: false
  isDefault: false
EOF
----
